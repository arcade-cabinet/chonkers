# Database

`src/persistence/sqlite/` is the **relational data layer** for chonkers. It uses `drizzle-orm` for typed schema and queries, ships a pre-built `public/game.db` as the initial database snapshot, and replays drizzle-kit-generated migration files forward against persisted user copies on subsequent runs.

`@capacitor/preferences` (the kv side, `src/persistence/`) handles small JSON values. This document is exclusively about the relational side.

## Layer model

```
                    ┌────────────────────────────────┐
                    │  src/persistence/sqlite/schema/  (drizzle TS)  │
                    │  Source of truth for shape     │
                    └───────────────┬────────────────┘
                                    │ drizzle-kit generate
                                    ▼
                    ┌────────────────────────────────┐
                    │  drizzle/0000_*.sql, ...       │
                    │  Forward-only migration ladder │
                    │  Committed to git              │
                    └───────────────┬────────────────┘
                                    │
            ┌───────────────────────┼────────────────────────────┐
            │                       │                            │
            ▼ build time            ▼ test time                  ▼ user runtime
   scripts/build-game-db.mjs   makeTestDb() factory     src/persistence/sqlite/bootstrap.ts
   (better-sqlite3, Node)      (better-sqlite3, Node)   (capacitor-sqlite)
            │                       │                            │
            ▼                       ▼                            ▼
   public/game.db            in-memory or file        first run: import asset
   shipped with the app      (env-overridable)        subsequent: replay missing
                                                       migrations against persisted
```

### Build time (Node, `scripts/build-game-db.mjs`)

`prebuild` / `predev` / `pretest:browser` runs `scripts/build-game-db.mjs`. The script:

1. Instantiates a fresh `better-sqlite3` in-memory database.
2. Applies every committed migration in `drizzle/*.sql` in order.
3. Sets `PRAGMA user_version = N` where `N` is the count of applied migrations (the highest migration index + 1, or any monotonic ladder we adopt — see "Versioning" below).
4. Optionally seeds static reference data (analytics aggregate row stubs, AI profile keys as a lookup table — see schema below for what gets seeded).
5. Serialises the database via `db.serialize()` and writes the bytes to `public/game.db`.

The output `public/game.db` is **gitignored**. It's a derived artefact rebuilt deterministically from the migration ladder. CI rebuilds it before every test and build run.

### Test time (Node, `makeTestDb()` factory)

`src/persistence/sqlite/__tests__/test-db.ts` exposes `makeTestDb()`. The factory:

1. Resolves the database location:
   - Explicit `path: string` argument → write to that path (creates parent dirs).
   - Otherwise, env var `CHONKERS_TEST_DB_DIR` set → write to `${dir}/${test-id}-${uuid}.db` for diagnostic inspection after the run.
   - Otherwise (the default) → `:memory:`, throwaway with the connection.
2. Instantiates `better-sqlite3` against the resolved location.
3. Applies every committed migration in `drizzle/*.sql` in order.
4. Wraps it with `drizzle-orm/better-sqlite3` and returns a typed handle.

Each call returns a fresh isolated DB. No shared state, no cleanup hooks needed. In-memory dies with the connection. On-disk artefacts under `CHONKERS_TEST_DB_DIR` are deliberately left after the run so a failed test's exact DB state can be inspected with `sqlite3 path/to/foo.db`.

### User runtime (browser/Capacitor, `src/persistence/sqlite/bootstrap.ts`)

On first run:

1. Fetch `public/game.db` as bytes.
2. Hand the bytes to `@capacitor-community/sqlite`'s `importFromJson` flow (or its newer asset-import path), materialising a per-user OPFS-backed copy.
3. Mark the user's persisted copy with the same `PRAGMA user_version` as the served `game.db`.

On subsequent runs:

1. Open the persisted user copy via `@capacitor-community/sqlite`.
2. Read its `PRAGMA user_version` (call this `persisted_version`).
3. Read the served `public/game.db`'s expected version (we ship a `game-db.meta.json` alongside `game.db` containing `{ user_version: N, generated_at: "..." }` — generated by the same build script that produces `game.db`, so the runtime never has to deserialise the asset just to read its version).
4. If `persisted_version === served_version`: open as-is, proceed.
5. If `persisted_version < served_version`: replay migrations `drizzle/${persisted_version + 1}_*.sql` through `drizzle/${served_version}_*.sql` against the persisted copy in order, set `PRAGMA user_version = served_version`, proceed. The replay runs inside an outer transaction so a failure rolls back rather than leaving the user's DB half-migrated.
6. If `persisted_version > served_version`: refuse to start. The user's DB is from a newer version of the app than the bundle they're running. This shouldn't happen (app stores enforce monotonic versions), but if it does it indicates a downgrade attack or a bundle/data mismatch.

The runtime never executes `CREATE TABLE` on its own. Schema construction happens at build time only. The runtime's only schema operations are the version-gated forward replay of existing migration SQL.

### Why not run drizzle-kit generate at runtime?

We do not. Drizzle-kit is a build-time tool. Migration generation happens in CI/dev, the resulting SQL files commit to git, and only the SQL replays at runtime. The runtime has no drizzle-kit dependency.

## Versioning

Migration files are named `drizzle/NNNN_short_description.sql` where `NNNN` is a monotonic four-digit prefix starting at `0000`. drizzle-kit generates these from the schema diff.

`PRAGMA user_version` stores the integer count of applied migrations (or, equivalently, the highest applied migration index + 1). This is SQLite's built-in 32-bit int slot — no custom metadata table needed.

Migrations are **forward-only**. There is no `down()` half. To roll a schema change back, author a new forward migration that undoes it.

A CI gate runs `drizzle-kit check` to verify that the committed migrations match the current schema (no schema-drift between TS and SQL).

## Tables

Three tables in the initial migration `drizzle/0000_initial.sql`:

### `matches`

One row per played game.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | `text` | PK | UUID v4 |
| `started_at` | `integer` | notNull, default `Date.now()` | epoch-ms |
| `finished_at` | `integer` | nullable | epoch-ms; null = in-progress |
| `winner` | `text` | nullable | `'red' \| 'white' \| 'forfeit-red' \| 'forfeit-white' \| null` (null = in-progress; the `forfeit-*` values record which player forfeited and trigger the game-over sting + the corresponding voice line on the *other* side) |
| `red_profile` | `text` | notNull | profile key from `docs/AI.md` (e.g. `aggressive-medium`), or `human` for hot-seat |
| `white_profile` | `text` | notNull | same |
| `opening_position_hash` | `text` | notNull | Zobrist hash at ply 0 — supports replay determinism |
| `coin_flip_seed` | `text` | notNull | the entropy that decided who went first; recorded for deterministic replay |
| `chain_source_col` | `integer` | nullable | active forced split chain source column, or null if no chain active |
| `chain_source_row` | `integer` | nullable | same, row |
| `chain_remaining_json` | `text` | nullable | JSON array of contiguous-run slice-index arrays still owed by the player; null if no chain active |
| `ply_count` | `integer` | notNull, default 0 | denormalised for analytics-join speed; updated on each move insert |

**Indices:**
- `idx_matches_finished_at` on `finished_at` — analytics queries by completion time.
- `idx_matches_profiles` on `(red_profile, white_profile)` — matchup analytics.

### `moves`

One row per ply.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `match_id` | `text` | notNull, FK→`matches.id` ON DELETE CASCADE | |
| `ply` | `integer` | notNull | 0-indexed |
| `color` | `text` | notNull | `'red' \| 'white'` |
| `from_col` | `integer` | notNull | 0..8 |
| `from_row` | `integer` | notNull | 0..10 |
| `to_col` | `integer` | notNull | 0..8 |
| `to_row` | `integer` | notNull | 0..10 |
| `slice_indices_json` | `text` | nullable | JSON int[] for split moves; null for full-stack moves |
| `stack_height_after` | `integer` | notNull | resulting stack height at `to`, 0..N (chonking replaces "capture") |
| `position_hash_after` | `text` | notNull | Zobrist hash of the full board after this move |
| `move_duration_ms` | `integer` | notNull, default 0 | how long the AI spent thinking (or human spent deciding); useful for difficulty calibration |
| `created_at` | `integer` | notNull, default `Date.now()` | |

**Primary key:** `(match_id, ply)`.

`moves` rows are inserted **live during the match** — one INSERT per ply. The capacitor-sqlite plugin coalesces a single statement into its own auto-transaction, so per-ply inserts are atomic and crash-safe. Per-move write volume is trivial at human pace and AI pace alike.

### `ai_states`

Latest AI dump snapshot per match per profile. Replace-not-append: a new dump for the same (match_id, profile_key) overwrites the previous row.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `match_id` | `text` | notNull, FK→`matches.id` ON DELETE CASCADE | |
| `profile_key` | `text` | notNull | `red_profile` or `white_profile` from the matches row, or `human` |
| `ply` | `integer` | notNull | the ply this snapshot was taken at |
| `dump_blob` | `blob` | notNull | output of `dumpAiState()` — opaque to persistence |
| `dump_format_version` | `integer` | notNull | for future-proofing the dump format; currently `1` |
| `created_at` | `integer` | notNull, default `Date.now()` | |

**Primary key:** `(match_id, profile_key)`.

Resume reads exactly one row per AI: `SELECT dump_blob, ply, dump_format_version FROM ai_states WHERE match_id = ? AND profile_key = ?`. There's no scan, no multi-row coalescence; it's a primary-key point-read.

For human players (`profile_key = 'human'`) there's no dump_blob to store, so the row is never written. The schema permits it for symmetry but the UI never produces one.

### `analytics_aggregates`

Materialised aggregate rows. Refreshed by the sim broker on match-end (and on match-forfeit). One row per `aggregate_key`.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `aggregate_key` | `text` | PK | e.g. `winrate:aggressive-hard:vs:defensive-easy`, `avg_ply_count:overall`, `forfeit_rate:by_profile:defensive-medium` |
| `value_json` | `text` | notNull | JSON-encoded aggregate value (number, object, array, depending on the key family) |
| `sample_count` | `integer` | notNull | how many matches contributed to this aggregate |
| `last_match_id` | `text` | nullable, FK→`matches.id` ON DELETE SET NULL | the most recent match this aggregate counted |
| `refreshed_at` | `integer` | notNull, default `Date.now()` | |

The aggregate-key naming convention is `family:dimension1:dimension2:...`. The sim broker owns the catalogue of aggregate keys and the refresh logic per key family. Adding a new aggregate is a code change in `src/analytics/`, not a schema change.

**Why materialised, not on-demand?** At rc-stage 10,000 matches, a "win rate by profile pair" query that scans `matches` with GROUP BY runs in tens of milliseconds — fine for an admin debug screen, slow for a live HUD. Materialised rows are sub-millisecond. The sim broker amortises the refresh cost over match-end events (already a write-heavy boundary).

Initial migration seeds zero rows. Aggregates accumulate as matches finish.

## What is NOT in the database

- **Settings (volume, mute, last-camera-angle, last-used profile, tutorial-seen, prefers-reduced-motion override)** → `kv` (`src/persistence/`).
- **AI profile *definitions*** (the 9 profile keys with their force-multiplier weights) → code constants in `src/ai/`. The `red_profile` / `white_profile` columns in `matches` are foreign references to those code constants, validated at insert time. We do not maintain a `profiles` table; if we ever ship user-defined profiles, that's a schema migration at that point.
- **Asset paths, fonts, palette tokens** → code constants under `src/utils/manifest.ts` and `src/design/`.

## Repos

`src/store/repos/` exposes typed CRUD per table:

- `matchesRepo.ts` — `createMatch`, `finalizeMatch`, `forfeit`, `setChain`, `clearChain`, `incrementPly`, `getMatch`, `listMatches`, etc.
- `movesRepo.ts` — `appendMove`, `listMovesByMatch`, `latestMoveByMatch`.
- `aiStatesRepo.ts` — `upsertDump`, `getDump`.
- `analyticsRepo.ts` — `upsertAggregate`, `getAggregate`, `listByFamily`.

Repos take the drizzle handle as their first argument so callers can compose them inside transactions (drizzle's `db.transaction(tx => ...)`). No singleton shortcuts; the broker passes the handle.

## Concurrency

drizzle's `db.transaction(tx => ...)` is the only transaction primitive. The runtime adapter delegates to `@capacitor-community/sqlite`'s native transaction support; the build-time and test-time adapters delegate to `better-sqlite3`'s. Per-DB write serialisation is enforced inside the broker (a write-lock queue keyed on match_id) — concurrent writers never enter the same transaction simultaneously.

Reads are concurrent and lock-free.

## Test strategy

The test pyramid for `src/persistence/sqlite/` is:

- **Node tier (the bulk):** schema correctness, migration forward-replay determinism, repo CRUD, query result shapes, transaction semantics. Backed by `makeTestDb()` (in-memory `better-sqlite3` by default; on-disk under `CHONKERS_TEST_DB_DIR` for diagnostics). Fast, parallelisable, no browser.
- **Browser tier (thin):** the import-from-asset bootstrap path, the version-detection-and-replay flow, OPFS persistence smoke. Just enough to prove the runtime adapter works under real `@capacitor-community/sqlite`. Everything *meaningful* (CRUD, query shapes, transaction semantics) is already covered by the Node tier.

Both tiers exercise the *same* drizzle schema and the *same* migration files. There are no schema-only-in-tests artefacts.

No mocks. Each layer's tests use the real layer below it. The 100/1000/10000 broker-and-governor runs at alpha/beta/rc are the closure of integration testing.

## Future migrations

When schema changes are needed, the workflow is:

1. Edit `src/persistence/sqlite/schema/*.ts`.
2. Run `pnpm drizzle-kit generate`. drizzle-kit emits `drizzle/NNNN_*.sql`.
3. Review the generated SQL. Hand-edit if drizzle-kit's emit is non-optimal or carries a destructive change we want to soften.
4. Commit the schema edit and the generated SQL together.
5. The next `pnpm build` regenerates `public/game.db` with the new migration applied.
6. The next user runtime detects `served_version > persisted_version` and replays the new migration.

The migration ladder grows monotonically. We never edit a committed migration after it has shipped to users.
